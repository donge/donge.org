# 深度还原：某嵌入式系统 PAK 文件格式逆向与解密修复

在嵌入式开发和逆向工程中，我们经常会遇到各种自定义的资源打包格式（PAK）。最近我们分析了一个基于 ARM926EJ-S 处理器的固件系统，发现其资源文件（视频、音频、配置）被封装在加密的 `.pak` 容器中。

通过对固件 `story.axf` 进行逆向分析，我们成功还原了其加密算法，并修复了市面上流传的“解密不完全”的问题。

---

## 1. PAK 文件布局 (File Layout)

该 PAK 格式是一个典型的“头部 + 索引表 + 数据块”结构。所有字段均采用 **Little-Endian（小端序）** 存储。

### 1.1 文件结构图解

```text
+---------------------------------------+ 0x00
| Header (32 bytes)                     |
+---------------------------------------+ 0x20
| Offset Table                          |
| (EntryCount * 8 bytes, 64-bit Offsets)|
+---------------------------------------+ 0x20 + (EntryCount * 8)
| Padding / Alignment (Optional)        |
+---------------------------------------+ Entry 0 Offset
| Encrypted Data - Entry 0              |
+---------------------------------------+ Entry 1 Offset
| Encrypted Data - Entry 1              |
+---------------------------------------+ ...
```

### 1.2 头部 (Header) 详细定义

| 偏移 | 长度 | 名称       | 描述                                          |
| :--- | :--- | :--------- | :-------------------------------------------- |
| 0x00 | 4    | Magic      | 固定为 `PACK`                                 |
| 0x04 | 4    | Version    | 目前分析的版本为 `3`                          |
| 0x08 | 4    | EntryCount | 包内包含的文件数量                            |
| 0x0C | 4    | HeaderSize | 通常为 `0x20`                                 |
| 0x10 | 4    | TotalSize  | 整个 PAK 文件的总大小                         |
| 0x18 | 4    | XOR Seed   | **关键参数**，解密用的种子（如 `0xFEDCBA98`） |
| 0x1C | 4    | Checksum   | CRC-16/CCITT 校验和                           |

---

## 2. 核心算法：循环密钥流解密

通过逆向固件中的 `pack_read` 函数，我们定位到了位于偏移 `0x1602C` 的解密逻辑。

### 2.1 算法原理

这是一种基于 **256 字节静态密钥表** 的流加密。每个字节的解密逻辑如下：

$$DecryptedByte = EncryptedByte \oplus (KeyTable[index] \oplus XOR\_Byte)$$

其中：

- `index`: 为该字节在当前 Entry 中的相对位置（从 0 开始），与 255 取模循环。
- `XOR_Byte`: 取自 Header 0x18 处的 Seed 低 8 位（例如 `0x98`）。
- `KeyTable`: 固化在固件中的 256 字节随机表。

### 2.2 逆向出的 ARM 汇编逻辑

```arm
0x1603C: ADD R4, R2, R12      ; R2=起始位置, R12=当前偏移i
0x16040: AND R4, R4, #0xFF    ; 计算索引 index = (pos + i) & 0xFF
0x16048: LDRB R4, [R5, R4]    ; 从 KeyTable[index] 加载密钥字节
0x1604C: EOR R4, R4, R3       ; 密钥字节 ^ XOR_Byte (0x98)
0x16050: EOR R14, R14, R4     ; 加密数据 ^ 组合密钥
```

## 3. 为什么之前的解密工具失效了？

在对比实验中，我们发现了一个有趣的现象：市面上某些解密后的 MP4 文件无法在标准播放器中打开。

### 3.1 错误现象 (Degradation)

通过分析 `wrong_decrypted.mp4`，我们发现其解密密钥流在 **40 字节（0x28）** 之后发生了退化：

| 数据位置      | 错误解密器密钥       | 正确解密器密钥           | 结果         |
| :------------ | :------------------- | :----------------------- | :----------- |
| 0x00 - 0x27   | `KeyTable[i] ^ 0x98` | `KeyTable[i] ^ 0x98`     | **部分匹配** |
| **0x28 以后** | **`0x98` (常量)**    | **`KeyTable[i] ^ 0x98`** | **解密失败** |

### 3.2 对 MP4 格式的影响

由于解密在 40 字节后失效，导致 MP4 的关键结构被破坏：

- **ftyp Box**: 品牌信息位移。
- **free Box**: 彻底丢失（8 字节）。
- **mdat Box**: 偏移错误且长度计算不准。
- **moov Box**: 位于文件尾部的媒体索引数据完全无法识别。**没有 `moov`，MP4 就无法寻址播放。**

---

## 4. 修复后的 MP4 布局对比

修复算法后，解密出的 Entry 0 呈现出完美的 MP4 结构：

| Box 类型 | 偏移 (Correct) | 偏移 (Wrong) | 状态             |
| :------- | :------------- | :----------- | :--------------- |
| **ftyp** | 0x0000         | 0x0000       | OK               |
| **free** | 0x0020         | 缺失         | **ERROR**        |
| **mdat** | 0x0028         | 0x0020       | **偏移错误**     |
| **moov** | 0x1C33D        | 无法解析     | **导致无法播放** |

---

## 5. 结论

该 PAK 格式的安全性主要依赖于固件内置ের **256 字节 KeyTable**。如果解密脚本没有提取该表，或者仅处理了文件头部，就会导致后续媒体数据依然处于半加密状态。

**正确的实现关键点：**

1. 提取完整的 KeyTable（位于固件 `0x254BC`）。
2. 使用 `(index & 0xFF)` 循环遍历 KeyTable。
3. 确保每个 Entry 的解密计数器在开始处重置为 0。

通过修复此算法，我们成功提取了该系统内的所有 20 个视频、654 个 MP3 音频以及所有 UI 资源图片。
