## 缘起

最近半年我经常做的一件小事：连 ClickHouse 看一眼某个表里有什么数据。

打开 DBeaver 太重，`clickhouse-client` 进交互模式后还要 `SHOW DATABASES` → `USE xxx` → `SHOW TABLES` → `SELECT * FROM yyy LIMIT 100`，鼠标键盘倒腾半天。其实我想要的就是：

```sh
ls /mnt/ch/db/default/
head /mnt/ch/db/default/access_local/all.tsv
wc -l /mnt/ch/db/default/access_local/20240504.tsv
```

像浏览本地目录一样浏览 ClickHouse。于是我用一个周末写了 **[ClickFS](https://github.com/donge/clickfs)**：一个 Rust 写的 FUSE 文件系统，把任意 ClickHouse 服务挂载成只读的 POSIX 目录树。

整个工程一千多行 Rust，一个静态二进制，Linux 和 macOS 都能跑。在线 demo 和文档都在 [donge.github.io/clickfs](https://donge.github.io/clickfs/)。

## 路径映射

ClickFS 把 ClickHouse 的对象层级直接映射到目录：

```
/mnt/ch/
└── db/
    └── <database>/
        └── <table>/
            ├── all.tsv              # 整张表，TSV 流式读出
            ├── <partition>.tsv      # 单个分区
            └── .schema              # SHOW CREATE TABLE 输出
```

设计上有几个故意：

- **`all.tsv` 用 TSV 而不是 CSV**：ClickHouse 自己最常用的是 `FORMAT TSV`，逗号在数据里太常见，转义麻烦
- **`.schema` 而不是 `schema.sql`**：跟 `.git` 一样以点开头，不污染 `ls` 默认输出
- **没有 `.parts/` 之类的元目录**：用户视角只有"库 → 表 → 分区"三层，不暴露 MergeTree 内部
- **整个 `/mnt/ch` 第一层就是 `db/`** 而不是直接铺开数据库名：留出未来加 `/mnt/ch/queries/` 之类只读虚拟目录的空间

## 流式读，不缓存

文件系统接口的隐含假设是"文件大小是固定的、可以随机寻址"。但 ClickHouse 表可能上 TB，把整张表 buffer 到内存里显然不行。

ClickFS 的取舍：**所有 `*.tsv` 文件都报告一个伪造的 8 EB 大小，主流操作以顺序读为主**。

- 第一次 `read(fd, off=0)` 时，向 ClickHouse 发起 `SELECT * FROM table FORMAT TSVWithNames`，把响应的 chunked HTTP body 当成 byte stream
- 后续 `read(fd, off=N)` 如果 `N == 之前累计读出的字节数`，沿用同一个流；否则视为 seek，重起一次查询
- 用户 `head -100` 提前关闭 fd → 我们 drop 掉那个 reqwest stream → 底层 HTTP 连接关闭 → ClickHouse 服务端的 `max_execution_time=60` 兜底，查询被中断

### tail / less 怎么办

最初的版本对反向 seek 直接返回 `EIO`，因为不想为了 `tail` 把整张表缓存或者按 `OFFSET` 一遍遍重发查询。

后来发现一个轻量办法：**tail 时按主键倒排扫描**。

`tail -n 100 all.tsv` 实际只想看最后 100 行。我们识别这种 "在文件末尾附近 read" 的访问模式，发出的查询变成：

```sql
SELECT * FROM table ORDER BY <primary_key> DESC LIMIT 100 FORMAT TSVWithNames
```

ClickHouse 对主键倒序扫的代价跟正序基本一样（MergeTree 的 sparse index 支持双向），不需要全表扫。然后我们在内存里把这 100 行**反转回正序**喂给用户，`tail` 看到的就是正常顺序的最后几行。

`less` 跳到末尾、`Shift+G` 之类也走同一条路径。这样一来：

- ✅ `head`、`cat | grep`、`awk` 顺序消费
- ✅ `head -1000` 在 70 万行的表上立刻返回，不会扫全表
- ✅ `tail -100`、`less +G` 也能跑了，代价是一次主键倒序的小查询
- ❌ `wc -c` 整个文件会因为 8EB 假大小直接打脸（`wc -l` 顺序读没问题，只是会扫全表）
- ❌ `less` 在文件中部任意 seek 仍会导致重发查询，体验略卡

## 为什么是只读

写操作（`open(O_WRONLY)`、`mkdir`、`unlink`）一律返回 `EROFS`。

不是技术做不到，是**语义不清**：

- `echo "1,2,3" >> table.tsv` 应该是 `INSERT INTO table VALUES (1,2,3)` 还是追加到某个分区？
- `rm 20240504.tsv` 应该是 `ALTER TABLE DROP PARTITION` 吗？那 `unlink` 失败了怎么回滚？
- 如果支持创建文件，那目录权限怎么映射 ClickHouse 的 GRANT？

POSIX 文件系统的事务模型和 ClickHouse 的 DDL/DML 模型完全对不上。强行做映射只会写出难以预期的工具。**只读，纯粹观察**，让 ClickFS 只解决"浏览"这一个问题。

## 实现要点

### FUSE 库选择

用了 [fuser](https://github.com/cberner/fuser)。它在 Linux 上有纯 Rust 的 ABI 实现（不依赖 libfuse），macOS 上需要装 [macFUSE](https://osxfuse.github.io/) 然后链接 framework。比 `fuse-rs`（停更）和老的 `fuse`（绑定 libfuse2）都活跃。

### 异步与同步的桥

FUSE 回调（`read`、`readdir`）是同步的，每次回调在 worker thread 里被调用。但 reqwest 是异步的。我用了一个常驻的 `tokio::Runtime`，FUSE 回调里 `rt.block_on(future)` 把异步操作同步化。

代价是每次 read 都要切到 tokio 线程池再回来，但比起 ClickHouse 一次查询的几十毫秒，这点开销可以忽略。

### 句柄复用

`open()` 时启动一个 `StreamHandle`，里面持有：

```rust
struct StreamHandle {
    bytes_read: u64,
    pending: BytesMut,           // 已收到、尚未给到 read() 的 chunk
    rx: mpsc::Receiver<Bytes>,   // 后台 task 写入的 channel
    cancel: oneshot::Sender<()>,
}
```

后台 task 把 reqwest stream 拆成 `Bytes` 往 channel 里灌，前台 read 调用 `rx.recv()` 拼到 buffer。drop 时 `cancel` 触发后台 task 退出，连带 drop reqwest，HTTP 连接随之关闭。

### TTL 元数据缓存

`ls /mnt/ch/db/default/` 会触发 FUSE 的 `readdir`，要去 ClickHouse 查 `SHOW TABLES`。如果用户在 shell 里 `ls`、`tab补全`、`find` 连环用，每次都打 ClickHouse 太蠢。

加了一层 TTL 2 秒的内存缓存（库列表、表列表、分区列表、存在性探测各一份），用 `dashmap` 做并发结构。2 秒既能让连续操作命中缓存，又不会把刚 `CREATE` 的新表藏太久。

## 跨平台编译

支持 Linux x86_64、Linux aarch64、macOS arm64 三个平台的预编译二进制。Linux 用 musl 静态链接（不挑 glibc 版本，Amazon Linux 2 那种 glibc 2.26 老系统也能跑），macOS 必须动态链接 macFUSE framework（macFUSE 是内核扩展，无法静态化）。

GitHub Actions 矩阵编译，打 `v*` 标签时自动产出 release。我最初矩阵里加了 macOS x86_64（Intel mac）一项，但 GitHub 的 macos-13 Intel runner 排队动辄 30 分钟，最终去掉了。Intel mac 用户通过 Rosetta 2 跑 arm64 binary 完全没问题，install.sh 里做了自动 fallback。

## 安装与使用

一句话装：

```sh
curl -fsSL https://donge.github.io/clickfs/install.sh | sh
```

或者从 crates.io：

```sh
cargo install clickfs
# 或者预编译版本
cargo binstall clickfs
```

挂载：

```sh
CLICKFS_PASSWORD=secret clickfs mount http://your-clickhouse:8123 /mnt/ch &
ls /mnt/ch/db/
```

卸载：

```sh
fusermount3 -u /mnt/ch    # Linux
umount /mnt/ch            # macOS
```

## 它适合什么场景

**适合**：

- 临时探索一个陌生的 ClickHouse 实例（"这库里都有些什么表？"）
- 写 shell pipeline 处理 ClickHouse 数据（`cat all.tsv | awk | sort | uniq -c`）
- 排查时快速 `head` 几行看 schema 和实际值
- 把 ClickHouse 数据丢给只认文件的工具（比如某些老 ETL）

**不适合**：

- OLTP 式的随机点查（用 SQL 客户端）
- 大量并发读（每个 read 都是一次 HTTP 查询）
- 需要写入或修改数据（永远不会支持）
- 需要秒级感知 schema 变化（受 TTL 缓存影响）
